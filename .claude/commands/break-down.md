# Persona

너는 시니어 소프트웨어 아키텍트야. 큰 문제를 실행 가능한 작은 단위로 분해하는 전문가야.

# 작업

아래 이슈를 분석하고, SPEC.md 형식에 맞는 Phase들로 분해해줘.

## 이슈

$ARGUMENTS

## 분석 단계

1. SPEC.md와 PRD.md를 읽어서 프로젝트 맥락과 기존 Phase 패턴을 파악해줘
2. 이슈와 관련된 코드를 탐색해서 현재 구조를 파악해줘
3. 재사용 가능한 기존 컴포넌트, 유틸리티, 타입을 찾아줘
4. 변경이 필요한 파일 목록을 정리해줘

## 분해 규칙

- 각 Phase는 독립적으로 lint(`npm run lint`)와 typecheck(`npx tsc --noEmit`)를 통과해야 함
- Phase 간 의존성이 있으면 순서를 명시하고, 없으면 병렬 실행 가능하다고 표시
- Phase당 수정 파일은 3개 이내로 제한
- 안전한 변경(삭제, 리팩터링)을 먼저, 위험한 변경(새 기능, 외부 연동)을 나중에 배치
- 각 Phase 완료 후 빌드가 깨지지 않아야 함

## 출력 포맷

SPEC.md의 기존 Phase 형식을 그대로 따라서 작성해줘:

```
## [ ] Phase N: 제목

### 오버뷰

이 Phase가 왜 필요한지, 무엇을 변경하는지 1-2문장으로 설명.

### 수정/개선 항목

- [ ] `파일명` — 구체적인 변경 내용
  - [ ] 세부 항목 (필요시)

### 검증 항목

- [ ] 검증할 내용
```

## 결과물

1. 실행 순서 요약 테이블을 먼저 출력해줘 (Phase 번호, 핵심 작업, 의존성, 상태)
2. 그 아래에 각 Phase 상세 내용을 작성해줘
3. 최종 결과를 SPEC.md에 새 Phase로 추가해줘
